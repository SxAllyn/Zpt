# 安全开发实践

本文档描述 Zpt-core 项目的安全要求和最佳实践，重点关注自研协议栈的安全性。

## 1. 安全原则

### 1.1 最小权限原则
- 代码仅拥有完成功能所需的最小权限
- 文件系统访问限制在必要目录
- 网络访问遵循白名单原则

### 1.2 纵深防御
- 多层安全控制，单一防线失效不影响整体安全
- 输入验证、访问控制、加密保护等多重机制
- 关键操作需要多重确认

### 1.3 不信任原则
- 所有外部输入均不可信
- 内部组件间也应有适当边界
- 默认拒绝，明确允许

## 2. 密码学安全

### 2.1 自研协议安全要求

#### Ztp 协议
- 帧完整性保护：HMAC-SHA256
- 重放攻击防护：时间戳+序列号
- 密钥派生：HKDF-SHA256，足够熵的盐值

#### Zap 协议
- 认证方法安全性：
  - PSK：至少 256 位随机密钥
  - Token：单次使用，足够熵
  - TOTP：RFC 6238 标准实现
- 密钥交换：ECDHE-P256，前向安全
- 会话密钥：定期轮换，默认 24 小时

#### Zop 协议
- 混淆算法：避免降低加密强度
- 指纹伪装：不泄露真实身份信息
- 形态切换：平滑过渡，避免模式识别

### 2.2 加密算法选择
- 对称加密：AES-256-GCM，ChaCha20-Poly1305
- 非对称加密：ECDHE-P256，ECDHE-P384
- 哈希函数：SHA-256，SHA-384
- 密钥派生：HKDF-SHA256，Argon2id（密码衍生）
- 随机数生成：`crypto/rand`，拒绝系统随机源

### 2.3 密钥管理
```go
// 密钥管理接口示例
type KeyManager interface {
    // 生成新密钥对
    GenerateKeyPair() (publicKey, privateKey []byte, err error)
    
    // 安全存储密钥
    StoreKey(keyID string, key []byte) error
    
    // 安全获取密钥
    GetKey(keyID string) ([]byte, error)
    
    // 密钥轮换
    RotateKeys() error
}
```

## 3. 代码安全

### 3.1 输入验证
- **协议数据**：验证长度、格式、取值范围
- **网络数据**：验证源地址、包大小、频率限制
- **配置文件**：验证完整性、权限设置
- **命令行参数**：验证合法性、路径安全性

### 3.2 内存安全
- 缓冲区边界检查
- 避免使用 `unsafe` 包
- 释放资源（连接、文件描述符等）
- 使用安全的数据结构

### 3.3 并发安全
- 竞态条件检测：`go test -race`
- 锁顺序一致性
- Channel 正确关闭
- 超时和取消机制

## 4. 网络安全

### 4.1 TLS 配置
```go
// 安全 TLS 配置示例
func secureTLSConfig() *tls.Config {
    return &tls.Config{
        MinVersion: tls.VersionTLS13,
        CurvePreferences: []tls.CurveID{
            tls.X25519, tls.CurveP256, tls.CurveP384,
        },
        CipherSuites: []uint16{
            tls.TLS_AES_128_GCM_SHA256,
            tls.TLS_AES_256_GCM_SHA384,
            tls.TLS_CHACHA20_POLY1305_SHA256,
        },
        // 证书验证
        InsecureSkipVerify: false,
        // 会话票据
        SessionTicketsDisabled: false,
        // 客户端认证
        ClientAuth: tls.RequireAndVerifyClientCert,
    }
}
```

### 4.2 抗探测措施
- **流量分析抵抗**：恒定流量模式，随机填充
- **时序分析抵抗**：随机延迟，抖动引入
- **指纹抵抗**：动态 TLS 指纹，协议模拟
- **主动探测抵抗**：静默丢弃，错误响应模拟

### 4.3 DoS 防护
- 连接数限制
- 请求频率限制
- 资源配额管理
- 慢速攻击防护

## 5. 依赖安全

### 5.1 第三方库审核
- 仅使用活跃维护的库
- 审查库的安全历史记录
- 评估库的权限需求
- 定期更新依赖版本

### 5.2 漏洞管理流程
```bash

```

## 6. 安全审计

### 6.1 代码审计
- **静态分析**：使用 CodeQL、Semgrep、gosec
- **动态分析**：模糊测试、渗透测试
- **人工审查**：重点关注自研协议实现

### 6.2 协议审计
- 邀请密码学专家审查协议设计
- 第三方安全公司渗透测试
- 漏洞奖励计划（待建立）

### 6.3 审计清单
- [ ] 所有加密操作使用标准库
- [ ] 随机数生成正确使用 `crypto/rand`
- [ ] 密钥管理安全，无硬编码密钥
- [ ] 输入验证完整，无缓冲区溢出
- [ ] 错误处理不泄露敏感信息
- [ ] 日志记录不包含密钥等敏感数据
- [ ] 配置文件权限限制
- [ ] 网络通信加密完整
- [ ] 依赖库无已知漏洞
- [ ] 测试覆盖安全关键路径

## 7. 应急响应

### 7.1 漏洞报告流程
1. **接收报告**：security@example.com（待确定）
2. **确认漏洞**：72小时内确认漏洞
3. **修复开发**：根据严重程度制定修复计划
4. **发布修复**：提供安全更新
5. **披露协调**：适当时间公开漏洞信息

### 7.2 严重程度分级
- **严重**：远程代码执行、密钥泄露
- **高危**：认证绕过、信息泄露
- **中危**：拒绝服务、权限提升
- **低危**：信息泄露、配置错误

### 7.3 回滚机制
- 保持向后兼容性
- 支持配置版本迁移
- 提供降级指南
