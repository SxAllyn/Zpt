# Zpt-core 里程碑1（Ztp协议核心）完成阶段详细开发规划

## 一、规划背景

基于《开发规划.md》和《项目文档.md》，本阶段目标为**完成里程碑1全部内容**，即Ztp协议核心功能完整实现并通过验收标准。

### 当前状态分析
1. **已完成**：项目骨架、构建系统、开发规范、内核框架、Ztp帧格式基础、TCP传输层

   - 测试覆盖率≥60%目标

## 二、阶段目标与范围

### 核心目标
1. **功能完整性**：实现Ztp协议所有核心功能（流控、优先级、ACK、多路复用）

- **本阶段不包含**：Zap认证、TLS集成、混淆协议（这些属于里程碑2）

## 三、详细任务分解

### 第一周：流控核心机制实现

#### 任务1.1：ACK帧格式设计（1天）
- **输入**：研究项目文档中的帧格式定义
- **输出**：ACK帧payload格式设计文档
- **技术决策**：
  - **选项A**：扩展帧头添加序列号字段（破坏兼容性）
  - **选项B**：使用现有Payload字段（推荐，8字节：`uint32 AckedBytes + uint32 WindowSize`）
- **建议选择**：选项B，保持帧格式兼容

#### 任务1.2：流状态管理结构（2天）
- **文件修改**：`internal/protocol/ztp/session.go`
- **新增结构**：
  ```go
  type streamState struct {
      sentBytes   uint32    // 已发送未确认字节数
      ackedBytes  uint32    // 已确认字节数
      windowSize  uint32    // 当前窗口大小（字节）
      priority    uint8     // 优先级（0:普通, 1:高, 2:低）
  }
  ```
- **集成到Session**：`streams map[uint32]*Stream` 改为 `streams map[uint32]*streamContext`

#### 任务1.3：ACK生成与处理（2天）
- **发送方**：Stream.Write()检查窗口可用性，阻塞机制实现
- **接收方**：Session.handleDataFrame()消耗数据后自动生成ACK帧
- **ACK处理**：Session.handleAckFrame()更新发送方窗口状态
- **关键算法**：滑动窗口信用制，初始窗口=65535字节

### 第二周：优先级系统与流接受机制

#### 任务2.1：优先级标志位扩展（1天）
- **文件修改**：`internal/protocol/ztp/frame.go`
- **新增常量**：
  ```go
  const (
      FlagLowPriority FrameFlags = 1 << 3  // 低优先级标志
  )
  ```
- **优先级映射**：
  - 高优先级：`FlagPriority = 1`，`FlagLowPriority = 0`
  - 普通优先级：`FlagPriority = 0`，`FlagLowPriority = 0`
  - 低优先级：`FlagPriority = 0`，`FlagLowPriority = 1`

#### 任务2.2：优先级发送队列（2天）
- **文件修改**：`internal/protocol/ztp/session.go`
- **方案选择**：
  - **选项A**：单个优先级队列（需排序，复杂）
  - **选项B**：多个优先级通道（高/普通/低三个通道，简单）
- **建议选择**：选项B，实现简单，性能可预测
- **发送逻辑**：优先从高优先级通道读取，再到普通，最后低优先级

#### 任务2.3：AcceptStream完整实现（1天）
- **机制设计**：远程STREAM_OPEN帧 → 流接受队列 → AcceptStream()获取
- **文件修改**：`internal/protocol/ztp/session.go`
- **新增字段**：`pendingStreams chan *Stream`
- **集成测试**：模拟客户端打开流，服务端AcceptStream接收

#### 任务2.4：流控与优先级整合测试（1天）
- **测试场景**：
  1. 窗口耗尽时Write阻塞
  2. 高优先级帧优先发送
  3. ACK更新后窗口恢复

### 第三周：回环测试与覆盖率提升

#### 任务3.1：回环测试组件实现（2天）

## 四、技术决策点汇总

| 决策点 | 选项A | 选项B | 建议选择 | 理由 |
|--------|-------|-------|----------|------|
| ACK帧格式 | 扩展帧头 | 使用Payload字段 | B | 保持帧格式兼容，实现简单 |
| 流控窗口单位 | 字节为单位 | 帧数为单位 | A | 更精确，与TCP兼容，符合"信用制"设计 |
| 优先级队列 | 单个优先级队列 | 多个优先级通道 | B | 实现简单，性能可预测，避免排序开销 |
| 回环测试 | 完整组件 | 简单集成测试 | A | 符合规划要求，创建可重用测试组件 |
| 序列号方案 | 显式序列号 | 隐式累积偏移 | B | 不修改现有帧格式，简化实现 |

## 五、风险与应对策略

### 高风险：流控实现复杂度
- **风险描述**：滑动窗口实现可能引入竞态条件和死锁
- **影响程度**：高（可能导致协议不稳定）
- **应对策略**：
  1. 简化初始设计（基础滑动窗口，非优化信用制）
  2. 使用atomic操作管理计数
  3. 早期加入竞态检测测试
  4. 分阶段实现：先实现阻塞，再优化性能

### 中风险：测试覆盖率难以达标
- **风险描述**：transport包从0%到80%需要大量测试代码
- **影响程度**：中（可能延误验收时间）
- **应对策略**：
  1. 优先编写关键路径测试（Dial、Read、Write、Close）
  2. 使用表格驱动测试减少重复代码
  3. 借助测试辅助函数简化测试编写
  4. 设定阶段性覆盖率目标（每周提升20%）

### 低风险：优先级队列性能影响
- **风险描述**：优先级队列可能成为发送瓶颈
- **影响程度**：低（可后续优化）
- **应对策略**：
  1. 使用高效通道选择（select优先高优先级）
  2. 基准测试验证性能影响
  3. 准备优化方案（如批处理发送）

### 外部风险：时间估算偏差
- **风险描述**：实际开发时间超出3-4周估算
- **影响程度**：中（影响整体项目进度）
- **应对策略**：
  1. 每周进度检查，及时调整计划
  2. 优先实现核心功能（流控 > 优先级 > 回环测试）
  3. 准备简化方案，确保基础功能完整
